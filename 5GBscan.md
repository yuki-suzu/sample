# 巨大ファイル（5GB超ZIP）スキャン機能 方式比較

本件は、大容量ファイル（解凍後数十GBのCSVを想定）のウイルススキャンを、既存のAPIレスポンス（即時スキャン）を劣化させることなく、いかに高速かつ安全にシステムへ統合するかを目的とした方式検討です。

## 案A：Shellスクリプト ＋ clamscan 方式（ファイル・バッチ処理型）

OS標準のコマンドラインツール（`unzip`, `split`）を利用してファイルを物理的に分割し、外部プロセスとしてスキャンを一括実行する方式です。

### 実装方針
1. JavaからShellスクリプトを呼び出し、ZIP内の大容量CSVをディスク上のテンポラリ領域に複数ファイルとして展開・分割する。
2. 分割された全ファイルに対し、`clamscan` を実行する。
3. スキャン完了後、テンポラリ領域のファイルを削除する。

### メリット (Pros)
*   **プロセスの完全分離**: API用の常駐スキャン機能（`clamd`）とは別のプロセスとして動くため、API側のスレッドを直接的に塞ぐことがない。
*   **初期開発コストが低い**: 複雑な並列処理やストリーム制御をプログラム内に記述する必要がなく、既存のコマンドの組み合わせで実現できる。

### デメリット (Cons)
*   **深刻なディスクI/Oボトルネック（性能劣化）**: 解凍後の数十GBものデータを一度ディスクに書き出し、それを再度スキャンエンジンが読み込むため、猛烈なディスクI/O（読み書き）が発生します。これにより、**CPUやメモリに余裕があっても、システム全体のレスポンス（他APIの応答速度等）がストレージの速度に引きずられて著しく低下する**リスクがあります。
*   **「ゴミファイル」の運用リスク（保守性低下）**: 処理の途中でエラーやサーバー再起動が発生した場合、数十GBの一時ファイルがディスク上に取り残されるリスク（容量圧迫）があり、確実なクリーンアップ機構などインフラ側での運用監視コストが増大します。

## 案B：Kotlin ＋ clamdscan (INSTREAM) 方式（オンメモリ・ストリーム型）

Java 21 / Spring Boot 3.2 のストリーム処理能力と Virtual Threads を活かし、ディスクを一切使わずにメモリ間の通信のみでスキャンを完結させるモダンな方式です。

### 実装方針
1. Kotlin側で大容量ZIPをストリームとして開き、少しずつメモリ上のバッファ（数MB）に読み込む。
2. ディスクには一切書き出さず、チャンク（分割単位）ごとにローカルの `clamd` へ直接TCP通信（`INSTREAM`）でデータを流し込む。
3. Java 21の Virtual Threads を用いて「API用スレッドの余力」の範囲内で並列ストリーム処理を行い、高速に処理を完了させる。

### メリット (Pros)
*   **ディスクI/Oゼロによる圧倒的な高速化（性能）**: 巨大な一時ファイルの「書き込み・読み込み」という最大のボトルネックが完全に消滅します。処理速度が「ディスクの速度」から「CPU・メモリの速度」へ引き上げられるため、スキャン完了までの時間が劇的に短縮されます。
*   **完全なクリーンステート（保守性・安全性）**: 検査対象のデータはメモリ上を通過するだけ（使い捨て）であるため、異常終了時でもサーバー上に「消し忘れの巨大ファイル」が残るリスクが物理的に存在しません。運用監視の手間が大幅に削減されます。
*   **モダンな技術資産の獲得**: Java 21 の最大の目玉である「Virtual Threads」とストリーム処理を組み合わせたこのアーキテクチャは、リソース効率が極めて高く、今後の大容量データ処理（数十GB〜100GB）にもそのまま横展開できる堅牢な基盤となります。

### デメリット (Cons)
*   **初期実装コストの増加**: TCPソケット通信の制御、ストリームの分割ロジック、およびAPIをブロックしないためのスレッド流量制御など、Kotlin側での高度な実装力とテスト工数が必要になります。

---

### 【総括・アーキテクト視点での推奨】

本システムにおいて、数十GB規模に膨らむ大容量ファイルを「物理ファイルとしてディスクに読み書きする（案A）」というアプローチは、ストレージI/Oを逼迫させ、結果的にシステム全体のスループットを低下させる旧時代的なボトルネックとなります。

開発の初期工数は若干増加しますが、最新スタック（Java 21 / Spring Boot 3.2）のポテンシャルを最大限に引き出し、**「他処理を邪魔せず最速で終わる」「運用中の予期せぬディスク枯渇事故を防ぐ」ことができる【案B（Kotlin＋オンメモリ方式）】の採用を強く推奨します。**

これはハードウェアリソース（メモリやディスク容量）の大小に関わらず、システムの健全性と拡張性を維持するためのアーキテクチャ上の最適解です。
